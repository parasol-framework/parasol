[[appendix-migration-from-lua]]
[appendix]
== Migration from Lua

This appendix provides a systematic guide for developers migrating existing Lua 5.1/5.2 code to Tiri.  Each section identifies a breaking change, explains the rationale, and shows the equivalent Tiri construct.

[[sec-migration-equality]]
=== Equality and Inequality Operators

Lua's `==` operator is replaced by the `is` keyword.  The `~=` inequality operator is replaced by `!=`.

[source,tiri]
----
-- Lua
if x == nil then ... end
if y ~= 0 then ... end

-- Tiri
if x is nil then ... end
if y != 0 then ... end
----

The semantics are identical to Lua's equality: two values are equal if they have the same type and the same value, or if a `__eq` metamethod returns true.  See <<ch-operators>> for the complete operator reference.

[[sec-migration-operators]]
=== Changed Operators

[discrete]
==== Exponentiation

In standard Lua, `^` is the exponentiation operator.  In Tiri, `^` is bitwise XOR and `**` is exponentiation.

[source,tiri]
----
-- Lua
result = 2 ^ 10       -- 1024

-- Tiri
result = 2 ** 10      -- 1024
result = 0xFF ^ 0x0F  -- bitwise XOR: 240
----

[discrete]
==== Bitwise Operators

The `bit.*` library is removed.  Tiri provides native C-style bitwise operators on 32-bit integers:

[cols="2,3"]
|===
|Operator |Description

|`~x`     |Bitwise NOT
|`a & b`  |Bitwise AND
|`a \| b` |Bitwise OR
|`a ^ b`  |Bitwise XOR
|`a << b` |Left shift
|`a >> b` |Right shift
|===

[source,tiri]
----
-- Lua (with bit library)
local masked = bit.band(flags, 0xFF)
local shifted = bit.lshift(1, 4)

-- Tiri
masked = flags & 0xFF
shifted = 1 << 4
----

Compound bitwise assignment operators (`&=`, `|=`, `^=`) do not exist.  Use `x = x & y` instead.

[[sec-migration-indexing]]
=== Zero-Based Indexing

Tiri uses zero-based indexing for tables, strings and arrays.  In standard Lua, the first element of a table is at index 1 and `string.sub()` starts at position 1.  In Tiri, the first element is at index 0.

[source,tiri]
----
-- Lua: first element is t[1]
-- Tiri: first element is t[0]
t = { 'a', 'b', 'c' }
print(t[0])   -- 'a'
print(t[1])   -- 'b'

-- Lua: string.sub(s, 1, 3)
-- Tiri: string.substr(s, 0, 3)
s = 'hello'
print(s:substr(0, 3))   -- 'hel'
----

This change affects all table constructors, `ipairs()` iteration, string functions and array access.  When migrating numeric `for` loops that iterate over tables, adjust the start index from 1 to 0 and the end condition accordingly.

[[sec-migration-scoping]]
=== Variable Scoping

In standard Lua, an undeclared variable is implicitly global.  In Tiri, all variables and functions are local by default.  Use the `global` keyword to declare globals explicitly.

[source,tiri]
----
-- Lua
counter = 0              -- implicitly global
function inc()
   counter = counter + 1 -- accesses global
end

-- Tiri
global glCounter = 0     -- explicitly global
function inc()
   glCounter += 1        -- accesses global
end
----

The `local` keyword remains available for explicit scope control and multi-variable declarations.

[source,tiri]
----
local a, b, c = 1, 2, 3
----

See <<ch-variables-and-scoping>> for the full scoping rules, including the `global` keyword and naming conventions.

[[sec-migration-exception-handling]]
=== Exception Handling

`pcall()` and `xpcall()` are deprecated.  Tiri provides structured exception handling through `try-except`:

[source,tiri]
----
-- Lua
local ok, err = pcall(function()
   risky_operation()
end)
if not ok then
   print('Error: ' .. err)
end

-- Tiri
try
   risky_operation()
except e
   print('Error: ' .. e.message)
end
----

`try-except` supports filtered handlers with the `when` clause to catch specific `ERR` codes, a `success` block that runs when no exception occurred, and optional stack trace capture with `try<trace>`.

The `io.*` library functions now throw exceptions on error instead of returning error codes.  Code that previously checked return values from `io.open()` or `file:read()` can be replaced with `try-except` blocks:

[source,tiri]
----
-- Lua
local file, err = io.open('data.txt', 'r')
if not file then
   print('Failed: ' .. err)
   return
end

-- Tiri
try
   file = io.open('data.txt', 'r')
except e
   print('Failed: ' .. e.message)
   return
end
----

See <<ch-error-handling>> for the complete exception handling reference, including `raise`, `check`, `error()` and filtered handlers.

[[sec-migration-control-flow]]
=== Control Flow Changes

[discrete]
==== The goto Statement

The `goto` statement and `::label::` syntax are removed.  The constructs that `goto` is commonly used for in Lua are replaced by dedicated statements:

[cols="2,3"]
|===
|Lua Pattern |Tiri Replacement

|Skip to next iteration
|`continue`

|Break out of a loop
|`break`

|Cleanup before return
|`defer`
|===

[source,tiri]
----
-- Lua: skip even numbers with goto
for i = 1, 10 do
   if i % 2 == 0 then goto continue end
   print(i)
   ::continue::
end

-- Tiri
for i in {0..10} do
   if i % 2 is 0 then continue end
   print(i)
end
----

[discrete]
==== The continue Statement

`continue` skips the remainder of the current loop body and advances to the next iteration.  It works in `for`, `while` and `repeat` loops.  See <<ch-control-flow>> for details.

[[sec-migration-string-functions]]
=== String Function Changes

[discrete]
==== Renamed Functions

`string.sub()` is renamed to `string.substr()`.  The parameters are the same, but indices are zero-based:

[source,tiri]
----
-- Lua
string.sub('hello', 1, 3)    -- 'hel'

-- Tiri
string.substr('hello', 0, 3) -- 'hel'
----

[discrete]
==== Removed Functions

The following Lua string functions are removed:

[cols="2,3"]
|===
|Function |Replacement

|`string.gsub()`
|`regex.replace()` or `string.replace()`

|`string.match()`
|`regex.search()`

|`string.gmatch()`
|`regex.scan()`

|`string.find()` with patterns
|`regex.search()` or `string.find()` (literal only)
|===

See <<sec-migration-pattern-matching>> for the pattern-to-regex migration guide.

[[sec-migration-pattern-matching]]
=== Migration from Lua Patterns

Lua patterns are deprecated in favour of POSIX-compatible regular expressions provided by the `regex.*` interface.  The following table maps common Lua pattern constructs to their regex equivalents:

[cols="2,2,3"]
|===
|Lua Pattern |Regex Equivalent |Description

|`%a`
|`[a-zA-Z]` or `[:alpha:]`
|Letters

|`%d`
|`\d` or `[0-9]`
|Digits

|`%w`
|`\w` or `[a-zA-Z0-9_]`
|Alphanumeric characters

|`%s`
|`\s` or `[ \t\n\r\f\v]`
|Whitespace

|`%l`
|`[a-z]`
|Lower-case letters

|`%u`
|`[A-Z]`
|Upper-case letters

|`%p`
|`[[:punct:]]`
|Punctuation

|`%c`
|`[[:cntrl:]]`
|Control characters

|`.`
|`.`
|Any character (identical)

|`%b()`
|No direct equivalent
|Balanced match (use recursive patterns or manual parsing)
|===

Capitalised Lua pattern classes (e.g. `%A`, `%D`) represent the complement set.  In regex, use `[^...]` or the capitalised shorthand (`\D`, `\W`, `\S`).

[discrete]
==== Quantifiers

[cols="2,2"]
|===
|Lua |Regex

|`*`  |`*`
|`+`  |`+`
|`-`  |`*?` (lazy)
|`?`  |`?`
|===

Lua's `-` quantifier (lazy match) maps to `*?` in regex.

[discrete]
==== Captures

Lua patterns use `()` for captures.  Regular expressions use the same syntax, but Tiri's `regex.search()` returns match objects rather than raw strings:

[source,tiri]
----
-- Lua
local year, month, day = string.match(date, '(%d+)-(%d+)-(%d+)')

-- Tiri
m = regex.search(date, [[\d+)-(\d+)-(\d+)]])
if m then
   year, month, day = m[0], m[1], m[2]
end
----

[discrete]
==== Anchors

Lua uses `^` and `$` for start-of-string and end-of-string anchors.  Regular expressions use the same anchors.

[discrete]
==== Common Migration Examples

[source,tiri]
----
-- Lua: strip whitespace
s = string.gsub(s, '^%s+', '')
s = string.gsub(s, '%s+$', '')

-- Tiri
s = s:trim()
-- Or with regex:
s = regex.replace(s, [[^\s+]], '')
s = regex.replace(s, [[\s+$]], '')
----

[source,tiri]
----
-- Lua: split on comma
for item in string.gmatch(csv, '([^,]+)') do
   process(item)
end

-- Tiri
items = csv:split(',')
for i, item in ipairs(items) do
   process(item)
end
----

[source,tiri]
----
-- Lua: replace all occurrences
result = string.gsub(text, 'old', 'new')

-- Tiri
result = text:replace('old', 'new')
-- Or with regex for pattern-based replacement:
result = regex.replace(text, [[pattern]], 'replacement')
----

See <<ch-regular-expressions>> and <<ch-strings>> for the complete API references.

[[sec-migration-removed-functions]]
=== Removed and Replaced Functions

The following standard Lua functions are removed or replaced:

[cols="2,3"]
|===
|Function |Replacement

|`loadstring(Code)`
|`load(Code)` — compiles and returns a function from a string.

|`dofile(Path)`
|`loadFile(Path)` — loads and executes a file.

|`select(N, ...)`
|Result filter operator `[mask]`.  See <<ch-result-management>>.

|`unpack(Table)`
|Removed.  Use multiple assignment or `table.unpack()` if available.

|`getfenv()` / `setfenv()`
|Removed.  Environment manipulation is not supported.

|`gcinfo()`
|Removed.  Use the garbage collector interface if memory reporting is needed.

|`table.maxn()`
|Removed.  Use `#t` for the length of a table or array.
|===

[[sec-migration-removed-libraries]]
=== Removed Libraries

The following standard Lua libraries are removed because their functionality is provided by the Kōtuku framework:

[discrete]
==== Package Library

The `package` library (`require`, `module`, `package.path`, etc.) is removed.  Tiri provides `include` and `import` for script organisation.  See <<ch-script-management>>.

[source,tiri]
----
-- Lua
local json = require('json')

-- Tiri
include 'json'
-- or
import 'json'
----

[discrete]
==== OS Library

The `os` library is removed.  Time, date and filesystem operations are provided by Kōtuku classes:

[cols="2,3"]
|===
|Lua Function |Kōtuku Equivalent

|`os.time()`
|Kōtuku Time class

|`os.date()`
|Kōtuku Time class

|`os.clock()`
|Kōtuku Time class

|`os.execute()`
|Kōtuku Task class

|`os.remove()` / `os.rename()`
|Kōtuku File class
|===

[discrete]
==== Debug Library

The introspective portions of the debug library are removed to reduce binary size.  A limited debug library is available for diagnostics.  See <<ch-the-debug-library>>.

[[sec-migration-arrays]]
=== Arrays as a Distinct Type

Lua uses tables for both associative maps and sequential data.  Tiri introduces arrays as a separate, performance-optimised type for sequential (integer-indexed) data.  Prefer arrays over tables when storing ordered, numerically indexed data.

[source,tiri]
----
-- Lua: sequential table
local items = { 'a', 'b', 'c' }
table.insert(items, 'd')

-- Tiri: use an array
items = array('a', 'b', 'c')
items:push('d')
----

Arrays are zero-indexed, support dedicated methods (`push`, `pop`, `map`, `filter`, `sort`, etc.) and offer better iteration performance than tables.  See <<ch-arrays>> for the complete API.

[[sec-migration-new-features]]
=== New Features with No Lua Equivalent

The following Tiri features have no counterpart in standard Lua.  They do not require migration but are available to improve code that was previously limited by Lua's feature set:

* **Compound assignment**: `+=`, `-=`, `*=`, `/=`, `%=`, `..=` (see <<ch-variables-and-scoping>>)
* **Postfix increment**: `counter++` (see <<ch-operators>>)
* **Arrow functions**: `n => n * 2` (see <<ch-functions>>)
* **String interpolation**: `f'Hello {name}'` (see <<sec-string-interpolation>>)
* **Safe navigation**: `obj?.field` (see <<ch-operators>>)
* **Ternary operator**: `cond ? a :> b` (see <<ch-operators>>)
* **If-empty operator**: `value ?? default` (see <<ch-operators>>)
* **Ranges**: `{0..10}`, `{0...10}` (see <<ch-ranges>>)
* **Type annotations**: `local x:num = 0` (see <<ch-type-system>>)
* **Constant variables**: `local x <const> = 42` (see <<ch-variables-and-scoping>>)
* **To-be-closed variables**: `resource <close> = ...` (see <<ch-resource-management>>)
* **Defer statement**: `defer ... end` (see <<ch-resource-management>>)
* **Choose expressions**: `choose x from ... end` (see <<ch-pattern-matching>>)
* **Result filter**: `[_*]func()` (see <<ch-result-management>>)
* **Pipe operator**: `x |> func()` (see <<ch-result-management>>)
* **Deferred expressions**: `<{ expr }>` (see <<ch-resource-management>>)
* **Thunk functions**: `thunk name():type ... end` (see <<ch-resource-management>>)
* **Unicode operators**: `≠`, `≤`, `≥`, `×`, `÷`, `↑` (see <<ch-operators>>)
