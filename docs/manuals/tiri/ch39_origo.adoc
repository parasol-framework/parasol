[[ch-origo]]
== Origo

Origo is the command-line executable that runs Tiri scripts.  It initialises the Kōtuku framework, parses command-line options, resolves the target script, passes arguments to the script environment, and manages the execution lifecycle.

On Linux systems the `tiri` symlink can also be used as an alternative.

[discrete]
[[sec-usage]]
=== Usage

[source,bash]
----
origo [options] [script.tiri] [arg1] [arg2=value] ...
----

If no script path is provided, Origo attempts automatic detection (see <<sec-automatic-script-detection>>).  Options are case-insensitive and begin with `--`.  Unrecognised options beginning with `--` are silently ignored.  All arguments after the script path are forwarded to the script as parameters accessible through `arg()` (see <<sec-script-parameters>>).

[discrete]
[[sec-commandline-options]]
=== Command-Line Options

[discrete]
==== General

[cols="2,4"]
|===
|Option |Description

|`--help`
|Prints usage information and exits.

|`--version`
|Prints version information on three lines — version number, git branch and commit hash, and build type — then exits.

|`--time`
|Prints the script execution time in seconds to stdout after the script completes.

|`--dialog`
|Displays an interactive file dialog for choosing a script to execute.

|`--statement <code>`
|Executes a single Tiri statement or expression instead of loading a script file.  The next argument is treated as the code to execute.

|`--procedure <name>`
|Specifies a procedure within the script to execute.

|===

[discrete]
==== Logging

All log output is directed to stderr.  Command-line flags control the minimum visible log level; each flag enables its level and all levels above it.  See <<sec-log-levels>> for the full level hierarchy and available flags.

When debugging thread-related issues, append `--log-threads` to include thread identification in log output.

[discrete]
==== Framework

[cols="2,4"]
|===
|Option |Description

|`--gfx-driver=<name>`
|Specifies the graphics driver instead of accepting the default.  Use `--gfx-driver=headless` for automated testing and CI environments where no display is available.

|`--set-volume <name>=<path>`
|Defines or overrides a filesystem volume at startup.  Can be specified multiple times.  Volume paths can reference other volumes.  See <<sec-virtual-volumes>> for further detail on volumes.

|`--no-crash-handler`
|Disables the default crash handler.  On Unix systems, this disables signal handlers for SIGINT, SIGTERM, SIGSEGV, and related signals.  Recommended when debugging with `gdb` and programs that may need to process these signals.

|`home=<name>`
|Customises the name of the home folder directory (default: `kotuku`).  This segregates user configuration and cache data.

|`--jit-options <options>`
|Passes comma-separated options to the JIT compiler (see <<sec-jit-options>>).
|===

[discrete]
[[sec-inline-execution]]
=== Inline Execution

The `--statement` option executes Tiri code directly without loading a script file:

[source,bash]
----
origo --statement "print('Hello World')"
----

[discrete]
[[sec-automatic-script-detection]]
=== Automatic Script Detection

When no script path is provided on the command line, Origo searches for a script to execute in the following order:

. If `package.zip` exists in the executable's directory, a `package:` volume is created from the archive and `package:main.tiri` is executed.
. If `main.tiri` exists in the current working directory, it is executed.
. If neither is found, the help message is displayed.

This feature makes it simple to distribute packaged Tiri applications by copying the Origo executable with a new name (e.g. `myapp.exe`) and including a `package.zip` or `main.tiri` file alongside it.  The user will now be able to execute the program by running `myapp.exe` without needing to specify commandline parameters.

[discrete]
[[sec-script-arguments]]
=== Script Arguments

Arguments after the script path are passed to the script and accessed through the `arg()` function (see <<sec-script-parameters>>).  Three forms are supported:

[cols="2,3,2"]
|===
|Form |Meaning |Access

|`flag`
|Boolean flag, set to `'1'`.
|`arg('flag')` returns `'1'`

|`key=value`
|Named parameter with a string value.
|`arg('key')` returns `'value'`

|`key={ item1 item2 }`
|Array parameter with indexed elements.
|`arg('key(0)')` returns `'item1'`; `arg('key:size')` returns the element count.
|===

All argument values are strings.  Use `tonumber()` for numeric conversion.

[source,bash]
----
origo myscript.tiri verbose width=1024 files={ a.txt b.txt }
----

[source,tiri]
----
if arg('verbose') then print('Verbose mode') end
width = tonumber(arg('width', '800'))   -- 1024
first = arg('files(0)')                 -- 'a.txt'
count = arg('files:size')               -- '2'
----

[discrete]
[[sec-shebang]]
=== Shebang Support

On Unix systems, Tiri scripts can be made directly executable by adding a shebang line:

[source,tiri]
----
#!/usr/bin/env origo
print('Hello from a Tiri script')
----

Mark the script as executable with `chmod +x script.tiri` to run it directly from the shell.

[discrete]
[[sec-exit-codes]]
=== Exit Codes

Origo returns `0` on successful execution.  A non-zero exit code indicates that the script or framework encountered an error.  If the script sets an error code or error message on its Script object, the corresponding error code is returned to the shell.

[discrete]
[[sec-version-output]]
=== Version Output

The `--version` option prints three lines:

[source,bash]
----
$ origo --version
Kotuku 2026.2.27 <1>
master:5edae466 <2>
Build Type: Debug <3>
----
<1> Framework version.
<2> The git branch and commit hash of the build.
<3> The build type (`Debug` or `Release`).
